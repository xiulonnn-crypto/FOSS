# 个人期权卖方助手后续产品规划

## Summary
- 规划方向：继续以 **Cash-Secured Put（CSP）** 为核心，把当前工具从“候选筛选 + 持仓雷达 + 复盘”升级为完整的个人决策闭环。
- 数据策略：**免费行情限定**，默认只围绕 `yfinance` 等免费源设计；不把 Tradier、Massive、IBKR 作为核心依赖。
- 输出颗粒度：偏产品方案，不展开具体代码接口和表结构；每期给出目标、用户流程、核心能力、验收标准和不做事项。
- 总路线：`免费行情可信度 → 期权池 → 智能开仓信号 → 智能平仓信号 → 盈亏复盘反哺`。

## Phase 1：免费行情可信度与数据分级
**目标**
- 解决免费行情“不稳定、延迟、字段缺失、链数据质量不一”的问题。
- 让用户知道哪些候选是“可决策”，哪些只是“参考”。

**用户流程**
- 用户保存观察名单并运行扫描。
- 系统展示每个标的和合约的数据状态：正常、延迟、缺字段、价差异常、链缺失、财报未知。
- 候选结果不再只显示分数，还显示“数据可信度”。

**核心设计**
- 增加数据质量分级：`A 可用于决策`、`B 可观察`、`C 数据不足`。
- 质量维度包括：现价时间、bid/ask 完整性、spread 合理性、OI/volume 是否缺失、IV/Greeks 是否供应商提供或 BS 估算、财报日期是否可得。
- 所有信号页面标注免费源限制：延迟、非专业实时、字段可能缺失。
- 扫描结果为空时给出具体原因，而不是只显示“无候选”。

**验收标准**
- 用户能一眼看出“为什么这个合约没进候选”。
- 数据异常不会被包装成高分开仓信号。
- 扫描结果能按数据质量筛选。

**不做**
- 不接付费行情。
- 不承诺实时性。
- 不做自动交易。

## Phase 2：期权池与观察池
**目标**
- 从“单次扫描结果”升级为“可持续跟踪的期权池”。
- 让用户能管理：哪些标的适合卖 Put，哪些合约值得等待，哪些机会已经失效。

**用户流程**
- 用户维护标的池。
- 系统扫描后生成合约池。
- 用户可把合约加入观察池，后续雷达持续跟踪。
- 合约从“观察中”变为“可开仓”或“失效”。

**核心设计**
- 三层池：
  - 标的池：AAPL、MSFT、SPY 等基础观察对象。
  - 合约池：每次扫描得到的 CSP 候选。
  - 观察池：用户手动关注但暂不入场的合约。
- 合约状态：`新发现`、`观察中`、`可开仓`、`已忽略`、`已过期`。
- 增加“忽略原因”：流动性差、收益不足、财报近、价格太近、主观不想做。
- 候选保留历史快照，用于后续复盘“错过的机会”。

**验收标准**
- 用户能把一个候选加入观察池并持续跟踪。
- 用户能看到一个合约为什么从可选变为不可选。
- 复盘中能区分“实际交易”和“观察但未交易”的机会。

**不做**
- 不做完整历史期权链回测。
- 不做多腿组合池。
- 不引入复杂组合保证金模型。

## Phase 3：智能开仓信号
**目标**
- 把当前线性评分升级为“可解释的开仓决策卡”。
- 不做黑盒预测，只做规则透明、可复盘、可调整的个人决策辅助。

**用户流程**
- 用户进入期权池。
- 系统给每个候选展示：建议开仓、等待、拒绝。
- 用户点开决策卡，查看收益、风险、数据质量和因子解释。
- 用户确认开仓后，系统记录当时的信号环境。

**核心设计**
- 开仓建议分三档：
  - `可开仓`：数据质量合格，收益、流动性、安全垫、DTE、Delta 均通过。
  - `等待`：方向可观察，但价格、IV、收益或安全垫未达标。
  - `拒绝`：存在硬性风险，如财报近、价差过大、OI 过低、Delta 过高。
- 决策卡展示：
  - 收益：权利金、年化收益、最大收益、资金占用。
  - 风险：安全垫、距离行权价、Delta、到期天数、财报距离。
  - 流动性：spread、OI、volume。
  - 波动：IV、RV 代理 IV Rank、是否为估算值。
  - 入场时机：RSI、布林带距离、近期下跌/反弹状态。
- 信号解释用自然语言，例如：“收益达标，但价差偏宽，建议等待更好报价”。

**验收标准**
- 每个开仓建议都有明确理由。
- 用户无需看代码也能理解评分来源。
- 开仓后能在复盘看到当时的决策卡快照。

**不做**
- 不输出“必买/必中”式预测。
- 不做机器学习模型。
- 不自动下单。

## Phase 4：智能平仓与持仓动作建议
**目标**
- 从“提醒有风险”升级为“给出持仓动作建议”。
- 帮助用户判断：继续持有、止盈、止损、防守、等待到期。

**用户流程**
- 用户进入持仓页。
- 每个持仓显示当前状态：健康、可止盈、需防守、到期风险。
- 系统给出动作建议和理由。
- 用户手动确认平仓或记录继续持有原因。

**核心设计**
- 平仓建议分为：
  - `止盈`：已捕获 50% 或 75% 最大盈利。
  - `加速止盈`：短时间内已捕获较高利润，建议锁定。
  - `时间退出`：剩余 14 天或 7 天，gamma 风险上升。
  - `防守`：Delta 突破、股价接近行权价、浮亏扩大。
  - `观察`：收益和风险都未触发动作。
- 每条建议展示触发条件、当前数值、阈值和严重程度。
- 用户平仓时可选择原因，并与最近信号关联。
- 对被行权持仓只做提示，不进入股票交易自动流程。

**验收标准**
- 用户能看到每个持仓“现在该不该动”的明确解释。
- 平仓原因与复盘切片一致。
- 同一信号不会重复刷屏。

**不做**
- 不自动滚仓。
- 不自动同步股票持仓。
- 不计算复杂税务或券商真实保证金。

## Phase 5：盈亏复盘与因子反哺
**目标**
- 从“看结果”升级为“知道哪些因子真的有效”。
- 让复盘能反过来调整筛选阈值和评分权重。

**用户流程**
- 用户完成若干笔交易后进入复盘页。
- 系统按因子切片展示胜率、ROE、MAE/MFE、平均持仓天数。
- 系统指出：哪些入场条件表现好，哪些条件导致浮亏大。
- 用户据此调整设置。

**核心设计**
- 复盘切片：
  - Delta 桶：0.10-0.15、0.15-0.20、0.20+。
  - DTE 桶：21-30、31-45、46-60。
  - IV Rank 桶：低、中、高。
  - RSI 桶：超跌、中性、过热。
  - 安全垫桶：小、中、大。
  - 出场原因：止盈、时间退出、防守、手动、到期、被行权。
- 增加“因子体检”：
  - 最赚钱的因子组合。
  - 最大浮亏最高的因子组合。
  - 胜率高但收益低的组合。
  - 少样本警告。
- 增加“设置建议”，例如：“近 20 笔中 Delta > 0.20 的 MAE 明显偏高，建议降低 delta_max”。

**验收标准**
- 用户能回答：什么条件下卖 Put 表现最好。
- 用户能看到开仓评分与最终盈亏是否相关。
- 复盘结论能转化为设置调整建议。

**不做**
- 不把样本不足的结论包装成确定规律。
- 不做全市场历史回测。
- 不做投资建议，只做个人历史统计。

## Phase 6：产品收口与长期扩展预留
**目标**
- 在 CSP 闭环稳定后，再考虑 Wheel 或多腿策略。
- 扩展前先确保个人使用体验稳定、解释清楚、复盘闭环完整。

**核心设计**
- CSP 完整闭环验收后，才灰显预留：
  - Wheel：被行权后 Covered Call。
  - Put Spread：降低资金占用。
  - Iron Condor：中性波动率策略。
- 这些只作为未来导航，不进入当前主流程。

**验收标准**
- CSP 用户流程不被未来策略复杂度干扰。
- 现有数据、复盘和信号命名能为后续策略扩展预留空间。

**不做**
- 免费行情阶段不优先做复杂多腿策略。
- 不做券商交易自动化。
- 不为了扩展牺牲当前 CSP 体验。

## Test Plan
- 场景验收：观察名单扫描、候选入池、观察池跟踪、开仓决策卡、持仓雷达、平仓建议、复盘切片。
- 数据异常验收：行情缺失、bid/ask 缺失、spread 异常、OI 缺失、财报日期未知、链为空。
- 决策解释验收：每个开仓/平仓建议必须能展示触发因子和阈值。
- 复盘验收：每笔交易能关联入场快照、平仓原因、MAE/MFE 和因子切片。
- 回归验收：现有 MVP 的筛选、持仓、事件、复盘不被破坏。

## Assumptions
- 用户选择：产品方案优先，不在本规划中展开具体 API、表结构或代码任务。
- 用户选择：免费行情限定，不把付费数据源纳入核心路线。
- 用户选择：CSP 优先，Wheel 和多腿策略只作为后期预留。
- 产品原则：人在回路、本地优先、信号可解释、不自动下单。


# 第一期技术方案：免费行情可信度与数据分级

## Summary
- 目标：在现有 `yfinance → screener → candidates → 前端表格` 链路中加入数据质量分级，避免免费行情缺失、延迟、估算字段被误当成强信号。
- 范围：只做 **免费行情质量识别、候选质量标注、扫描诊断、空结果原因解释**。
- 不改变策略范围：仍只支持 CSP 卖 Put。
- 不接入付费源，不做自动下单，不做完整历史期权链回测。

## Key Changes
- 候选质量等级：
  - `A`：可用于个人决策；报价、价差、OI、Delta、IV/RV 代理等关键字段可用。
  - `B`：可观察；核心报价可用，但存在估算或缺失风险，如 BS Greeks、IV Rank 缺失、财报未知。
  - `C`：数据不足；不进入候选表，只进入扫描诊断，例如无双边报价、价差过宽、Delta 不可得、链拉取失败。
- 候选质量输出字段：
  - `quality_grade`: `A | B | C | unknown`
  - `quality_score`: `0-100`
  - `quality_flags`: JSON 数组，存储数据问题代码。
  - `quote_age_seconds`: 免费源估算延迟，`yfinance` 默认按约 900 秒。
  - `greeks_source`: `provider | bs_fallback | missing`
  - `iv_rank_source`: `rv_proxy | missing`
- 扫描批次诊断：
  - 每个 scan run 记录符号级统计：拉取成功/失败、扫描到期数、合约数、候选数、A/B/C 数量、主要拒绝原因。
  - `/api/scan/latest` 和 `/api/scan/run/:id` 在 `run` 中返回 `diagnostics`。
- 前端展示：
  - 候选表新增“质量”列，显示 A/B/未评级 badge。
  - 表格顶部显示本轮扫描质量摘要：A/B 候选数量、失败标的数、主要过滤原因。
  - 空结果时展示具体原因，例如“多数合约因 DTE 不符 / OI 过低 / bid-ask 无效被过滤”。

## Implementation Changes
- 数据模型与迁移：
  - 给 `candidates` 增加质量字段：`quality_grade`、`quality_score`、`quality_flags`、`quote_age_seconds`、`greeks_source`、`iv_rank_source`。
  - 给 `scan_runs` 增加 `diagnostics TEXT`，存 JSON。
  - `init_database` 增加兼容迁移，旧库启动时自动补列；旧候选缺质量字段时 API 返回 `unknown`。
- 核心逻辑：
  - 新增纯函数模块负责质量评估，输入为原始合约、Greeks 回填后的合约、Quote、settings、provider 信息，输出质量等级、分数、flags。
  - 重构 CSP 筛选为“同一套过滤逻辑同时产出候选和拒绝原因计数”，避免评分逻辑与诊断逻辑分叉。
  - `score_csp_candidates` 保持兼容，内部复用新评估逻辑。
- 扫描 Job：
  - `run_screener` 对每个 symbol 收集诊断：quote 错误、expiration 数、chain 错误、contracts_seen、candidate_count、rejection_counts、quality_counts。
  - 写入候选时同步写质量字段。
  - 完成事件 payload 增加质量摘要，例如 `{ "quality_counts": {"A": 8, "B": 3}, "failed_symbols": 1 }`。
- API：
  - 保持现有 `schema: "scan_latest_v2"` 不改名，向后兼容新增字段。
  - 候选行额外返回 `data_quality` 嵌套对象，前端优先用嵌套对象；旧行用 flat 字段兜底。
  - `scan_specific` 成功时也返回质量字段；失败时返回 `diagnostics.blockers`，用于解释具体不可计算原因。
- 前端：
  - 候选表新增质量列和 tooltip。
  - `renderCandidates` 兼容无质量字段的旧候选。
  - 扫描空状态读取 `run.diagnostics`，展示前三个主要过滤原因。
  - 入场确认时把质量字段带入 `POST /api/positions`，并进入 `open_snapshot`，方便后续复盘。

## Quality Rules
- Info flags 不降级：`provider_delayed`、`iv_rank_proxy`。
- Warn flags 降低质量分：`greeks_bs_fallback`、`iv_rank_missing`、`earnings_unknown`、`volume_missing`、`wide_but_allowed_spread`。
- Blocker flags 不入候选，只进入诊断：`invalid_bid_ask`、`wide_spread`、`delta_missing`、`oi_below_min`、`dte_out_of_range`、`roi_below_min`、`margin_buffer_low`、`earnings_within_window`、`iv_rank_below_min`、`provider_error`。
- 评分规则：
  - 初始 100 分。
  - 每个 warn 按严重程度扣 5-15 分。
  - `>= 90` 且无 warn 为 A。
  - `60-89` 为 B。
  - blocker 或 `< 60` 为 C。
- `yfinance` 的免费延迟只作为 info，不自动降为 B。

## UI Copy
- A badge：`可决策`
- B badge：`可观察`
- unknown badge：`未评级`
- 空结果摘要示例：
  - `本轮扫描完成：无候选。主要原因：DTE 不在范围 42 条，OI 过低 18 条，价差过宽 9 条。`
  - `TSLA 期权链拉取失败；AAPL 有 3 条可观察候选但无 A 级候选。`
- 质量 tooltip 示例：
  - `免费延迟行情；Greeks 由 Black-Scholes 回填；IV Rank 为 RV 代理。`

## Test Plan
- 单元测试：
  - 有效双边报价、供应商 Greeks、RV 可用 → A。
  - BS 回填 Greeks 或 IV Rank 缺失 → B。
  - bid/ask 缺失、spread 超限、Delta 缺失 → C/blocker。
  - 拒绝原因计数与现有过滤阈值一致。
- Job 集成测试：
  - FakeProvider 正常返回候选时，候选写入质量字段。
  - FakeProvider 某个 symbol 抛错时，scan run 写入 diagnostics，扫描仍完成。
  - 全部合约被过滤时，candidate_count 为 0，但 diagnostics 包含 rejection_counts。
- API 测试：
  - `/api/scan/latest` 返回候选质量字段和 `run.diagnostics`。
  - 旧候选无质量字段时，API 返回 `unknown` 而不是报错。
  - `/api/scan/specific` 成功返回质量字段，失败返回 blocker diagnostics。
- 前端验收：
  - 候选表能显示 A/B/未评级。
  - 空结果能显示主要原因。
  - 入场后复盘 snapshot 能看到入场时质量字段。
- 回归测试：
  - 现有 settings、watchlist、manual scan、specific search、positions、review 流程保持可用。

## Assumptions
- 第一期不新增设置项，质量分级使用现有 filters 阈值和固定质量规则。
- 不把 C 级合约写入 `candidates`，只写入 scan diagnostics。
- 不改变 `scan_latest_v2` schema 名称，只做向后兼容字段扩展。
- `yfinance` 仍是唯一核心行情源，所有质量提示围绕免费源限制设计。


# 第二期技术方案：因子挖掘与因子实验室

## Summary
第二期目标是在现有系统因子库之上新增“因子实验室”，让用户能用安全表达式 DSL 创建、批量试验、快速诊断并保存个人因子。实现上不生成 Python 源文件，所有手工/挖掘因子以 DB 定义保存，运行时动态注册为 `FactorBase` 兼容对象，继续复用现有 `/api/factors`、因子计算、IC/IR 评测、相关性热力图和观察池治理。

默认第一期数据可信底座已完成：基本面字段具备 `as_of_date`，因子计算不会读取未来财报，估值/财报质量字段可被查询。

## Key Changes
- 新增 `backend/factor_lab/`：
  - `parser.py`：基于 Python `ast.parse(..., mode="eval")` 的白名单解析器，禁止 `eval`、属性访问、下标访问、导入、lambda、推导式。
  - `operators.py`：实现表达式算子。
  - `runtime.py`：把 DB 因子定义动态包装为 `FactorBase` 子类。
  - `screening.py`：快速体检、候选筛选、批量参数展开。
- 扩展 `backend/factors/registry.py`：
  - 新增 `register_dynamic_factor(cls)`。
  - `list_factors()` 合并 Python 系统因子与 DB 启用因子。
  - `get_factor_class()` 支持 `standard_id` 和 DB 因子主键别名。
- 新增 `backend/api/factor_lab.py`，挂载 `/api/factor-lab`。
- 新增前端页面 `/factor-lab`，不引入 Monaco/CodeMirror，首版用普通 textarea + 字段/算子插入器，减少依赖和打包复杂度。
- `/factors` 增加来源筛选：系统因子、手工因子、挖掘因子。

## DSL Design
表达式采用函数调用风格，字段名和参数名都是普通标识符：

```text
zscore(winsorize(return(close, lookback=252, skip=21), lower=0.01, upper=0.99))
```

首版字段白名单：
- 行情：`open`、`high`、`low`、`close`、`volume`
- 估值：`pe_ratio`、`pb_ratio`、`ps_ratio`、`dividend_yield`、`market_cap`
- 财务：`revenue`、`net_income`、`total_assets`、`total_equity`、`total_debt`、`capex`、`operating_cashflow`、`ebitda`、`book_value`

首版算子白名单：
- 时间序列：`return(field, lookback, skip=0)`、`lag(field, days)`、`ts_mean(field, window)`、`ts_std(field, window)`、`ts_rank(field, window)`、`decay_linear(field, window)`
- 截面处理：`rank(x)`、`zscore(x)`、`winsorize(x, lower=0.01, upper=0.99)`
- 中性化：`neutralize(x, by="market_cap")`、`neutralize(x, by="sector")`、`neutralize(x, by="market_cap,sector")`
- 数学运算：`+`、`-`、`*`、`/`、一元负号；除零结果置为 `NaN`

执行规则：
- 所有算子输入输出均为长格式 panel 上的 `pd.Series`，按 `(date, symbol)` 对齐。
- 时间序列算子按 `symbol` 分组滚动。
- 截面算子按 `date` 分组。
- `direction` 由用户显式选择，默认 `+1`。
- 表达式运行失败时返回结构化错误，不写入 `factor_values`。

# 第二期技术方案：期权池与观察池

## Summary
- 目标：把当前“单次扫描候选表”升级为可持续跟踪的三层池：`标的池 → 合约池 → 观察池`。
- 约束：继续免费行情限定，继续只做 CSP 卖 Put，不接付费源、不做自动下单、不做多腿策略。
- 设计原则：扫描结果仍保留为快照；期权池保存“合约最新状态”；观察池保存“用户意图和跟踪状态”。

## Data Model
- 复用 `watchlist` 作为标的池，补充字段：
  - `pool_status`: `ACTIVE | PAUSED | ARCHIVED`
  - `tags`: JSON array，例如 `["核心科技", "ETF"]`
  - `notes`: 用户备注
  - `last_scanned_at`
  - `last_candidate_count`
  - `last_pool_summary`: JSON，保存最近一次该标的扫描摘要
- 新增 `option_pool` 作为合约池：
  - 唯一键：`symbol + expiration + strike + right`
  - 字段：最新 bid/ask/mid、spot、iv、iv_rank、delta、dte、annualized_roi、spread_pct、margin_buffer、score、quality_grade、quality_flags
  - 生命周期字段：`first_seen_at`、`last_seen_at`、`last_scan_run_id`、`latest_candidate_id`
  - 状态：`NEW | ACTIVE | STALE | EXPIRED | BLOCKED`
- 新增 `option_watchlist` 作为观察池：
  - 关联 `option_pool.id`
  - 状态：`WATCHING | READY | IGNORED | OPENED | EXPIRED`
  - 用户字段：`watch_reason`、`ignore_reason`、`target_premium`、`target_score`、`target_margin_buffer`、`notes`
  - 系统字段：`created_at`、`updated_at`、`last_evaluated_at`、`last_signal`
- 兼容策略：
  - `candidates` 继续作为扫描快照，不替代。
  - `positions.open_candidate_id` 保持不变。
  - 从观察池入场时，仍创建 `positions`，并在 `open_snapshot` 记录 `option_pool_id`、`option_watchlist_id`。

## Backend Flow
- `job_screener` 每次扫描后同时做两件事：
  - 写入本轮 `candidates` 快照。
  - upsert 到 `option_pool`，刷新合约最新指标和状态。
- 合约状态规则：
  - 本轮扫描命中且未过期：`ACTIVE`
  - 连续多轮未出现但未过期：`STALE`
  - 到期日小于今日：`EXPIRED`
  - 数据质量为 C 或存在 blocker：`BLOCKED`
- 观察池评估：
  - 每轮扫描后，重算所有 `WATCHING` 合约。
  - 达到用户目标或系统入场过滤时，改为 `READY` 并写事件。
  - 合约过期则改为 `EXPIRED`。
  - 用户忽略则改为 `IGNORED`，不再自动唤醒。
- 事件规则：
  - 新高分合约进入合约池：低优先级 info。
  - 观察合约变为 READY：warn。
  - 观察合约过期或数据失效：info。
  - 同一观察项同一状态不重复发事件。

## API Design
- 标的池：
  - `GET /api/pool/underlyings`
  - `PATCH /api/pool/underlyings/:symbol`
  - `POST /api/pool/underlyings/:symbol/pause`
  - `POST /api/pool/underlyings/:symbol/archive`
- 合约池：
  - `GET /api/pool/options`
  - 支持查询参数：`symbol`、`status`、`quality_grade`、`min_score`、`min_dte`、`max_dte`
  - 返回合约最新指标、最近扫描时间、是否已加入观察池
- 观察池：
  - `GET /api/watch/options`
  - `POST /api/watch/options`
  - `PATCH /api/watch/options/:id`
  - `POST /api/watch/options/:id/ignore`
  - `POST /api/watch/options/:id/open`
- 入场兼容：
  - 从候选入场：沿用当前 `POST /api/positions`。
  - 从观察池入场：调用同一接口，但附带 `option_pool_id`、`option_watchlist_id`，后端写入 `open_snapshot`。

## Frontend Design

  - 标的池：管理 symbol、启用/暂停、标签、备注、最近扫描摘要。
  - 合约池：展示最新可选 CSP 合约，支持质量、分数、DTE、Delta、状态筛选。
  - 观察池：展示用户正在等待的合约和触发状态。
- 候选行新增操作：
  - `入场`
  - `观察`

  - 状态 badge：观察中、已达标、已忽略、已过期。
- 空状态文案：
  - 标的池空：提示先添加观察标的。
  - 合约池空：提示运行扫描或放宽过滤。
  - 观察池空：提示可从合约池点击“观察”。

## Jobs & State Rules
- `job_screener`：
  - 正常扫描后刷新 `option_pool`。
  - 根据最新池状态评估观察项。
  - 写 scan diagnostics 时同时记录合约池更新数量。
- `job_settlement`：
  - 不直接管理观察池，但过期日后观察项会由下一轮扫描或轻量维护逻辑标记 `EXPIRED`。
- 新增轻量维护逻辑：
  - 启动 worker 时或每日一次，将已过期 `option_pool` 和 `option_watchlist` 标记为 `EXPIRED`。
- 不新增单独进程，不新增外部依赖。

## Migration Plan
- `init_database` 添加非破坏性迁移：
  - 给 `watchlist` 补充标的池字段。
  - 创建 `option_pool`、`option_watchlist`。
  - 为 `option_pool` 建唯一索引：`symbol, expiration, strike, right`。
  - 为 `option_watchlist` 建索引：`status, option_pool_id`。
- 旧数据处理：
  - 旧 `watchlist.enabled=1` 映射为 `pool_status=ACTIVE`。
  - 已有 `candidates` 不反向批量迁移到 `option_pool`；从下一次扫描开始生成合约池。
  - 旧持仓不需要回填观察池信息。

## Assumptions
